<h2>레이아웃 - 특히 들여쓰기</h2>

깔끔한 레이아웃을 유지하면 코드를 훨씬 더 읽기 쉽게 만듭니다.
또한 버그의 원인을 찾는 데 도움이 됩니다.
예를 들어 루프가 제대로 들여쓰기가 되어 있지 않다면 특정한 부분이
어디에 속한 것인지 불분명할 것입니다.

저 <hl>next</hl> 문은 어느 <hl>foreach</hl>에 속해 있을까요?

<code lang="perl">
foreach my $a (@array1) {
foreach my $y (@array2} { }
if (cond) {
next;
}
}
</code>

아래의 것이 훨씬 더 읽기 쉽습니다:

<code lang="perl">
foreach my $a (@array1) {
    foreach my $y (@array2} {
    }
    if (cond) {
        next;
    }
}
</code>

완벽하지는 않지만 <a href="https://metacpan.org/pod/Perl::Tidy">Perl::Tidy</a>
모듈에 들어 있는 <hl>perltidy</hl>라는 툴을 써서 여러분들의 코드의 레이아웃을
개선할 수 있습니다.

<h2>변수 이름은 내용을 나타내도록</h2>

위 예제에서 <hl>@array1</hl> 안에는 무엇이 들어 있을까요? 우리는 이게 배열이라는
것을 이름 앞의 <hl>@</hl>를 보고 알 수 있습니다. 이 변수의 이름이 array1이라는
것은 그다지 큰 도움이 되지 않습니다. 변수의 이름을 지을 때는 그 변수의 내용을 드러낼
수 있는 이름을 지으세요. 실제 이름이 무엇일지는 프로그램의 문맥에 좌우되겠지만
<hl>@users</hl>나 <hl>@server_names</hl> 같은 이름이면 훨씬 좋을 것입니다.

루프 변수도 마찬가지입니다. <hl>$a</hl>나 <hl>$y</hl> 같은 것보다 더 길고
의미있는 이름들이 있을 겁니다.

<h2>$a와 $b는 사용하지 마세요</h2>

특별하게, <hl>$a</hl>와 <hl>$b</hl>는 
<a href="https://perlmaven.com/sorting-arrays-in-perl">sort</a> 함수에서 사용되는
특수 변수입니다.
이 변수를 다른 곳에 쓰지 마세요. 짧은 일회용 코드에도요!
그저 혼란스러워질 뿐이고 그 변수는 <hl>my</hl> 선언을 필요로 하지 않기 때문에
초보자들을 어리둥절하게 할 것입니다.

(어째서 Perl은 $x가 선언되지 않았다고 불평하면서 $a에 대해서는 불평하지 않죠...?)


<h2>$_ 없애기</h2>

Perl의 디폴트 변수 <hl>$_</hl>는 굉장합니다만, 일반적으로는 이것을
코드에 직접 적지는 말아야 합니다.
<a href="/the-default-variable-of-perl">디폴트 변수</a>의
중요한 점은 여러분이 그것을 직접 타이핑할 필요가 없다는 점입니다.

<hl>$_</hl>는 여러분들 코드에서 눈으로 볼 수는 없어야 합니다.

주요 예외는 
<a href="https://perlmaven.com/transforming-a-perl-array-using-map">map</a>,
<a href="https://perlmaven.com/filtering-values-with-perl-grep">grep</a>
그리고 유사한 함수들입니다. 여기에서는 직접 적지 않을 수 없습니다.

그 외의 곳에서는, 만일 여러분이 <hl>$_</hl>라고 타이핑해야 한다면, 그건 아마
좀 더 <b>의미있는 이름</b>의 변수를 직접 선언해서 쓸 시점이라는 뜻입니다.

<h2>언제나 strict 와 warnings를 사용하세요</h2>

언제나 <a href="/strict">use strict</a>와
<a href="https://perlmaven.com/installing-perl-and-getting-started">use warnings</a>를
사용하세요. 이것들은 여러분의 안전망입니다!

<h2>렉시컬 파일핸들과 3인자 open</h2>

아주 구식 교육:

<code lang="perl">
open FH, $filename or die;
</code>

신식 교육:

<code lang="perl">
open my $fh, '<', $filename or die;
</code>

더 신식 교육:

<code lang="perl">
open my $fh, '<:encoding(UTF-8)', $filename or die;
</code>

쿨한 교육:

<code lang="perl">
use Path::Tiny qw(path);
my $fh = path($filename)->openr_utf8;
</code>

<a href="https://metacpan.org/pod/Path::Tiny">Path::Tiny</a>를 사용하는 쿨한 방법들이
몇 가지 더 있습니다만, 어떤 이유로든 Path::Tiny를 사용하지 못하더라도, 렉시컬 변수
(my $fh)와 파라메터 3개를 사용하는 open을 쓰실 수는 있을 겁니다.

어떤 경우에도,
<a href="https://perlmaven.com/open-files-in-the-old-way">오래된 방법으로 파일을 열지는 마세요</a>.

<h2>직접적인 객체 표기(Direct object notation)</h2>

이것은 문제점이라기보다는 제가 싫어하는 부분입니다.

어떤 클래스의 오브젝트를 생성할 때 다음과 같이 적으세요:

<code lang="perl">
Module->new(param, param);
</code>

다음과 같이 간접적인 객체 표기(indirect object notation) 말고요:

<code lang="perl">
new Module(param, param);
</code>

모듈의 설명서에서 두번째 방법을 보여주고 있더라도 말이죠.

대부분의 경우 이 두 가지는 동일하기는 합니다만, 모든
스크립트/응용프로그램/프로젝트/회사에서 한 가지 방법을 쓰는 게 더 낫고, 그렇다면, 제발,
첫번째 방법을 쓰세요.

<h2>Perl::Critic 사용하기</h2>

<a href="https://metacpan.org/pod/Perl::Critic">Perl::Critic</a>은 굉장한
모듈입니다. 저는 이 모듈 작성자도 알고 있습니다. 이 모듈에는
<b>perlcritic</b>이라는 툴이 들어 있는데 파일 이름을 입력받아서
개선할 수 있는 부분들의 목록을 알려줍니다.
따로 설정할 수도 있습니다만 디폴트 설정(엄격한 정도 5레벨. "gentle")으로
시작해도 좋습니다.


<h2>객체 지향 프로그래밍</h2>

여러분의 코드가 OOP로 작성되어 있거나, bless를 사용하고 있거나,
접근자 생성기(accessor generator)를 쓰고 있다면, 
<a href="https://perlmaven.com/moo">Moo</a>나
<a href="https://metacpan.org/pod/Moose">Moose</a>를
사용하도록 이전함으로써 코드를 개선할 수 있을지 모릅니다.
