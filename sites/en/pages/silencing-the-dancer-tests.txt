A lot of that noise comes from the logging facilities of Dancer that in the development environment
which uses the <hl>environments/development.yml</hl> configuration file is set to at level <hl>core</hl>.
It might be useful during development, but I find that distracting during testing.

The solution I found was to use a separate environment for testing. I think this can be useful later
on if we want to set other special values during testing. For now I've just set the Dancer envrionment
to be "test" and to create <hl>environments/test.yml</hl> where I can set <hl>log: "warning"</hl>.

In order to tell the tests to use this environment variable we can add

<code lang="perl">
BEGIN {
    $ENV{DANCER_ENVIRONMENT} = 'test';
}
</code>

to the beginning of every test script.
It has to be in a <a href="/begin">BEGIN block</a> and it has to be before the <hl>use D2::Ajax;</hl> statement,
loading the main module of our application, in order for Dancer to use it.

<h2>Make sure no more warnings will go unnoticed</h2>

Before fixing this warning, let's make sure no new warnings will go unnoticed. We can add
an extra test that will check if there was any warning during the test run and fail the test-script it there was any warning.

If we load the <a href="http://metacpan.org/pod/Test::NoWarnings">Test::NoWarnings</a> module will capture all the warnings
during the test-run and report them as failure at the end. In order to use it we only need to load the module with a

<code>
use Test::NoWarnings;
</code>

statement, and increment the test counter by one.

<h2>Fixing the uninitialized value warning</h2>

The area that generated the waning is this route, specifically line 33 is the line where we call <hl>reverse</hl>.

<code lang="perl">
get '/api/v2/reverse' => sub {
    my $text = param('str');
    my $rev = reverse $text;
    return to_json { text => $rev };
};
</code>

The problem is that in some of the cases <hl>$text</hl> is not initialized. We can easily fix this by setting it to
the empty string if if was <a href="/undef-and-defined-in-perl">>undef</a> using the
<a href="/what-is-new-in-perl-5.10--say-defined-or-state">defined-or operator</a> that was introduced in perl 5.10.

The result is this:

<code lang="perl">
get '/api/v2/reverse' => sub {
    my $text = param('str') // '';
    my $rev = reverse $text;
    return to_json { text => $rev };
};
</code>

If we run <hl>make test</hl> again we get that all the tests pass again.
